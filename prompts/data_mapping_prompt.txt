Rules:
- Focus on the final forward-engineered state as the primary target for mapping.
- Use the initial target specification and initial target data dictionary as comparison context, not as the final source of truth.
- Reverse-engineered forward-engineered output and generated SQL/code evidence must be used when available.
- Produce evidence-based mappings only. Do not invent tables or fields that are not grounded in the inputs.
- Make the mapping professionally useful for migration planning and design assurance.
- Make column mapping the centerpiece of the output.
- Prefer one row per important legacy-to-final column mapping wherever practical.
- Reconciliation approach must be practical for migration execution, not generic advice.
- Include reconciliation guidance for core business entities, financial outputs, consent/compliance data, rule-driven values, and audit/underwriting records where relevant.
- Cross-reference source dictionary, initial target dictionary, legacy spec, target spec, reverse-engineered final code, reverse-engineered final SQL, and forward-engineered final collated spec together before deciding mappings.
- Normalise terminology, data types, and key strategies so equivalent concepts are aligned even when naming differs.
- Infer migration intent and classify structural changes such as rename, merge, split, type promotion, key strategy change, compliance addition, column removal, denormalisation, and stored-procedure promotion.
- Align all field descriptions to business-readable language that a migration architect or business analyst can review.
- Prepare the output so it can support downstream semantic gap analysis, lineage tracing, and migration planning.
- Distinguish clearly between:
  - fields directly carried forward
  - fields transformed or renamed
  - fields newly introduced in the final design
  - fields not represented in the final design
- Prefer explicit entity and field names over vague descriptions.
- When a field appears to be derived or split across multiple final-state fields, explain the transformation logic and implementation location.
- Reference data, lookup tables, tax catalogs, rule tables, consent records, workflow state tables, and audit tables should be treated as important mapping artifacts.
- If the final design introduces governance or traceability fields such as rule versions, audit metadata, eligibility results, underwriting identifiers, or breakdown components, capture them in new_final_state_fields.
- For field mappings, explicitly state whether each field is direct, transformed, derived, defaulted, or unmapped.
- For each field mapping, use `data_mapping_rule` to explain how the final target field is populated from source or derived logic.
- For transformation logic, identify source fields, target fields, logic, and where that logic is implemented in the final design.
- For reconciliation, define dataset pairs, business keys, comparison method, tolerance, and exception handling.
- Every important final-state table and every important migration-relevant target column should be represented through field mappings, reference-data mappings, new final-state fields, or reconciliation rules. Do not omit major structures.
- Read service, repository, API contract, model, DTO, pricing, validation, workflow, and schema artifacts to confirm actual field semantics and computed-column meaning.
- Extract and preserve allowed values where they are explicitly known from dictionaries, schemas, API contracts, models, or code.
- Use migration notes normalisation:
  - Rename: `Renamed from <source_column>`
  - Type change: `Type changed from <source_type> to <target_type>`
  - New column: `New column. <reason>`
  - Key strategy: `PK strategy changed from <old> to <new>`
  - Computed: `Computed via <function_name>; replaces <source_logic>`
  - Unchanged: use `null` semantics in notes where no change is needed; do not use empty-string placeholders
- Collapse duplicates across DDL, repositories, APIs, and models into one canonical mapping row per column concept wherever possible.
- Source traceability matters. Use the evidence sections to cite files, functions, tables, or artifacts where the mapping or transformation is grounded.
- Numeric values must be JSON numbers, booleans must be JSON booleans, and absent values must be JSON null semantics where appropriate.
- Self-check before returning:
  - mapped and unmapped counts should agree with the listed rows
  - transformation count should match the transformations array length
  - reconciliation rule count should match the reconciliation_approach array length
  - new final-state fields should only include fields with no true legacy equivalent
  - unmapped legacy fields should contain only fields not represented in the final state
- Keep output concise but specific enough for real migration design review.
