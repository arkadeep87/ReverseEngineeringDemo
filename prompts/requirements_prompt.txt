Rules:
- Generate draft requirements only.
- Do not assume business approval.
- Highlight ambiguities in open_questions_for_sme.
- Include acceptance criteria wherever possible.
- Keep the output review-friendly and precise.
- Use business-readable language that an SME or business stakeholder can review without needing code knowledge.
- Avoid low-level technical wording unless it is required to preserve clarity or traceability.
- Phrase requirements in terms of business behavior, validation outcomes, compliance expectations, user experience, and operational needs.
- Treat the legacy capability baseline plus the gap analysis as the modernization scope.
- If a legacy capability is missing in the current target, assume it needs to be built unless the inputs explicitly say it is out of scope.
- Do not assume the current target limitation defines the final future-state scope.
- Include requirements for missing country-specific logic, missing workflows, and missing controls when those are identified in the gap analysis.
- Define country-specific requirements explicitly when the legacy system supports country-specific behavior that is missing or incomplete in the target.
- Create separate, traceable requirement IDs for major country-specific capabilities where needed, rather than collapsing them into one generic requirement.
- Ensure country-specific requirements are specific enough to drive technical design and forward engineering later.
- For country-specific scope, do not write only a generic statement such as "support country-specific rules."
- Write explicit requirements for each affected country when the inputs identify country-specific differences.
- For each country-specific requirement, name the country and the exact missing business behavior, such as validations, pricing adjustments, tax treatment, product eligibility, field behavior, workflow branching, compliance rules, notices, or reference-data usage.
- Treat country-specific requirements as first-class functional requirements, not as side notes or assumptions.
- Create separate functional requirement entries for each supported country and each materially different country rule area when the legacy and gap inputs show that behavior.
- If Spain, Portugal, Italy, Germany, the United Kingdom, Switzerland, or other countries appear in the inputs, write separate requirements for each of those countries where rules differ.
- Include the exact business rule, formula, threshold, rate logic, tax handling, sequencing rule, or reference-data dependency whenever it can be recovered from the inputs.
- If multiple formulas or rule variants exist for one country, split them into multiple requirement entries instead of collapsing them into one generic country requirement.
- Write requirement titles and descriptions in a concrete requirement style, for example `Spain quote pricing shall follow the approved Spain pricing model`, rather than generic modernization wording.
- If multiple rule types are missing for the same country, either capture them in one detailed country requirement or create multiple country-specific requirements so the downstream design can implement them precisely.
- Make the acceptance criteria country-aware and testable, so an SME can confirm what must happen for Spain, Portugal, Italy, Germany, the United Kingdom, Switzerland, or any other country found in the inputs.
- Write the business_context as polished business prose, not terse notes. It should summarize the modernization objective, current-state issues, target-state intent, operational impact, and why the initiative matters.
- Every functional requirement must feel implementation-ready from a business perspective: title, requirement statement, rationale, source gap, and measurable acceptance criteria.
- Write requirement descriptions in formal BRD language using business wording such as `The system shall...`, `The solution shall...`, or `The process shall...`.
- Do not write placeholder or generic requirement text such as `support country-specific logic` unless the exact behavior is genuinely unknown.
- For each functional requirement, explain the expected business outcome and user or operational impact clearly enough for SME approval.
- When a business rule varies by country, product, threshold, tax rate, surcharge, discount, eligibility band, or effective date, write the requirement so downstream design can externalize that rule into configuration or reference data instead of embedding literals in application code.
- When the inputs show reference-data usage, lookup-table usage, configurable thresholds, configurable rates, or configurable sequencing, state that dependency explicitly in the requirement description or acceptance criteria.
- For pricing, tax, discount, surcharge, eligibility, and compliance rules that may change operationally, prefer wording that supports rule externalization, versioning, effective dating, precedence, and auditable rule traceability.
- Use professional requirement IDs:
  - Functional requirements: `F001`, `F002`, `F003`, ...
  - Non-functional requirements: `N001`, `N002`, ...
  - Compliance requirements: `C001`, `C002`, ...
  - Data requirements: `D001`, `D002`, ...
  - UI requirements: `U001`, `U002`, ...
  - API requirements: `A001`, `A002`, ...
  - Migration requirements: `M001`, `M002`, ...
- IDs must be uppercase and zero-padded to 3 digits.
- Functional requirement titles should sound professional and business-facing, for example `Germany quote taxation shall apply approved insurance duty and statutory health surcharge rules`.
- Include source_gap values that clearly describe the business gap being addressed.
- For non-functional requirements, include service quality expectations such as auditability, maintainability, performance, resilience, traceability, and operational support where relevant.
- For compliance requirements, clearly state the jurisdiction, control expectation, and business consequence of non-compliance.
- For data requirements, describe data persistence, traceability, migration, and reporting needs in professional business language.
- For UI, API, and migration requirements, focus on business intent and operational necessity rather than technical implementation detail.
- The review_notes should read like real delivery notes to stakeholders, not generic filler.
- The final document should feel complete, professional, and suitable for real project review without changing the JSON structure.
- Return JSON only.
