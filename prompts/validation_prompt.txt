## PRE-COMPARISON NORMALIZATION (MANDATORY)

Before identifying any validation differences, you MUST normalize and canonicalize source expectations and generated behavior:

1. Normalize syntax and structure:

   * Treat equivalent implementations as the same even when file structure differs
   * Ignore framework-specific or code-organization-only differences
   * Ignore naming-only differences unless naming affects behavior, mapping, or integration

2. Normalize terminology:

   * Map different variable names to the same business meaning
   * Standardize business concepts such as validation, pricing, tax, discount, workflow, persistence, compliance, and auditability

3. Extract and enrich each expectation with:

   * business capability
   * rule_type
   * business_intent
   * validation dimension (functional parity, rule parity, workflow parity, compliance parity, data parity, or technical design parity)

4. Create a canonical comparison identity for each expected behavior:
   (business capability + rule_type + business_intent + validation dimension)

## SEMANTIC EQUIVALENCE LOGIC (CRITICAL)

The generated target MUST NOT be reported as different if:

* Business intent is the same
* Material behavior is the same
* Approved design intent is preserved

Treat generated behavior as EQUIVALENT if differences are only:

* Syntax differences
* File structure differences
* Naming differences
* Framework or implementation-style differences
* Non-material decomposition across multiple files

Treat generated behavior as DIFFERENT only if:

* Required logic is missing
* Formula or threshold materially differs
* Validation behavior differs
* Workflow sequencing differs
* Persistence behavior differs
* Compliance or audit behavior differs
* Approved technical design intent is not reflected

---

## VALIDATION DECISION LOGIC

For each important source expectation:

1. Find the matching generated implementation or evidence in generated files
2. Compare it against:

   * legacy source behavior
   * approved requirements
   * approved technical specification

3. If generated implementation is materially aligned:

   * DO NOT report it as a difference

4. If generated implementation is missing, partial, or materially different:

   * Add an item to differences
   * Add a targeted remediation item to suggestions when useful

---

## FALSE POSITIVE PREVENTION (HIGH PRIORITY)

DO NOT report a difference if:

* Equivalent behavior exists in the generated target
* Differences are only structural or cosmetic
* The generated target preserves the approved modernization intent even if implementation differs

When uncertain:

* Prefer not reporting a difference unless the mismatch is material
* Use notes to explain uncertainty
